## Goal

Move WorldBet2026 from `better-sqlite3` (synchronous, single on-disk file) to
**self-hosted PostgreSQL** on the EC2 box. The code becomes fully
Postgres-compatible so a later move to a managed/multi-server Postgres is just a
connection-string change. **No behavior changes** â€” money/grading logic stays
byte-identical; all 185 tests must stay green.

## Motivation

Managed/hosted DB plan, multi-server scaling, concurrency/reliability, preference.
Self-hosting on the box first is a stepping stone: it makes the _code_ PG-ready
without yet decoupling the DB from the box.

## Key decisions

- **Host:** self-hosted Postgres 16 on the EC2 box. Standard `pg` driver + a
  `pg.Pool` over localhost. New env var `DATABASE_URL` replaces `DATABASE_PATH`.
- **Driver:** `drizzle-orm/node-postgres`. The Postgres driver is **async** â€”
  this is the dominant cost: ~30 files using the synchronous
  `.get()/.all()/.run()` API and synchronous `db.transaction()` convert to
  `await`, and their call chains become `async`.
- **Money columns â†’ `bigint` (mode: 'number')**: `stake_mmk`, `net_mmk`,
  `fee_mmk`, `daily_total_limit_mmk`, `referral_bonus_mmk`. JS `number` is exact
  to 2^53, far above the 1e9 stake ceiling, so arithmetic code is untouched.
  Scores, counts, `version`, `ball_q`, `price_c`, `*_pct` stay `integer`.
- **Timestamps stay `text` (ISO strings)**, exactly as today. The codebase passes
  and compares ISO strings everywhere; converting to `timestamptz` would force
  conversions across the money path and risk timezone bugs. Faithful migration =
  keep text. (Can revisit later as a separate change.)
- **PK identity:** `integer ... generated by default as identity` (not `serial`)
  so the one-time data import can insert explicit IDs, after which sequences are
  reset.
- **Transaction atomicity:** `placeBet`'s version-lock + stake-limit logic is
  preserved exactly inside `await db.transaction(async (tx) => â€¦)`, using
  `SELECT â€¦ FOR UPDATE` row locks on the line / match-day rows to retain the
  guarantee better-sqlite3 gave synchronously.
- **Unique-violation retry** (ticket_no) matches Postgres error code `23505`
  instead of the SQLite message string.
- **Tests:** **PGlite** (in-process Postgres, WASM) replaces the `:memory:`
  SQLite test DB. Hermetic and fast, same feel as today.
- **Migrations:** fresh Postgres-dialect baseline `0000`; old SQLite migrations
  archived under `drizzle/sqlite-archive/` (they cannot run on PG).
- **Data cutover:** one-time script reads the live `worldbet.db` (keeps
  `better-sqlite3` as a devDependency for the script only) and inserts all rows
  into Postgres in FK order, then resets identity sequences. Brief maintenance
  window; SQLite file retained as instant rollback.
- **Ops:** `backup.sh` â†’ `pg_dump` (keep 14-day retention + nightly cron). Boot
  scripts (`seed`, `create-admin`, `bootstrap-admin`, `regrade`) rewritten to the
  pg client. `worldbet.service` gains `After/Requires=postgresql.service`.

## Verification gates

1. `npm test` green on PGlite (185/185).
2. `npm run build` clean (tsc).
3. Manual end-to-end on staging DB: place bet â†’ grade â†’ settle â†’ reconcile.
4. Production cutover only after row-count + balance spot-check vs current
   dashboard. Keep `worldbet.db` for rollback.

## Out of scope

Managed/Neon hosting, multi-instance app servers, PgBouncer, timestamp type
change. All become trivial follow-ups once the code is PG-native.
